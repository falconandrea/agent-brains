## Quick Start

### 1. New project ‚Äî full setup

```bash
# Link agents & skills with a PROFILE (curated subset per tech stack)
./setup.sh /path/to/your-project laravel     # or: nextjs | astro | nodejs | full

# Scaffold .ai context structure (copy ‚Äî project-specific)
./scaffold.sh /path/to/your-project
```

Available profiles: `laravel`, `nextjs`, `astro`, `nodejs`, `full` (all skills).
Default (no arg) is `full`. See [Profiles](#profiles) below.

### 2. Link .opencode only (no .agents)

```bash
./setup.sh /path/to/your-project opencode   # link .opencode only (no .agents)
```

### 3. Existing project (with local configurations)

If your target project already has a physical `.agents` or `.opencode` directory (for example, skills generated by third-party tools like `laravel-boost` or custom configurations):

1. **Migrate existing skills/workflows**: Check if there are local configurations you want to preserve, and copy them to the corresponding folders in this repository first (e.g. to `.agents/skills/` in `agent-brains`).
2. **Remove the local physical folder** in the target project:
   ```bash
   rm -rf /path/to/your-project/.agents
   ```
3. **Run the link script** (with a profile, or `full` for everything):
   ```bash
   ./setup.sh /path/to/your-project laravel   # or: nextjs | astro | nodejs | full
   ```

### 4. Unlink

```bash
./unlink.sh /path/to/your-project
```

## How It Works

### Canonical skill store ‚Üí single source of truth

`.agents/skills/` in **this repo** is the only place where skills live. Projects
never install or update skills directly ‚Äî they consume via symlinks (see Profiles).
How you update depends on the tool:

- **`npx skills`** (mattpocock, anthropics, vercel, ‚Ä¶) runs **here**, in this repo ‚Äî
  it downloads from GitHub and needs no project context:
  ```bash
  npx skills@latest add <source>
  npx skills update
  ```
- **`php artisan boost:update`** (`laravel/boost`) must run **inside a Laravel project** ‚Äî
  it reads the project's composer/packages to generate project-relevant skills. Run it
  in any Laravel project whose `.agents/skills/` links back here; the write flows through
  the symlinks into this canonical store, so every other Laravel project sees it:
  ```bash
  cd /path/to/laravel-project
  php artisan boost:update
  ```

This keeps one canonical version shared across all projects.

### Profiles ‚Üí curated subset per project

A project does **not** get every skill. It gets only the skills listed in its
profile manifest. `.opencode` is still symlinked whole (agents are opt-in by
invocation, no always-on cost); only `.agents/skills/` is curated.

```
# full profile (escape hatch):
project-a/.agents        ‚Üí  symlink  ‚Üí  this-repo/.agents            (all skills)

# profiled mode (e.g. laravel):
project-b/.agents/       ‚Üí  real dir
project-b/.agents/skills/  ‚Üí  real dir of per-skill symlinks ‚Üí this-repo/.agents/skills/<name>
```

Each symlink points into the canonical store, so an edit/install here is
immediately visible in every project whose profile includes that skill. Re-running
`./setup.sh /path laravel` refreshes the symlinks (useful after editing a manifest
or installing a new skill and adding it to a profile).

### Project Context (.ai) ‚Üí Copy (project-specific)

```
this-repo/templates/.ai/  ‚îÄ‚îÄcopy‚îÄ‚îÄ‚Üí  project-a/.ai/  (independent)
                           ‚îÄ‚îÄcopy‚îÄ‚îÄ‚Üí  project-b/.ai/  (independent)
```

Each project gets its own copy. PRD, roadmap, DB schema are unique per project.

### Laravel Boost Skills

`php artisan boost:update` runs **in a Laravel project** (not in this repo ‚Äî it
needs Laravel context to detect packages and generate project-relevant skills).
Run it in any Laravel project whose `.agents/skills/` is linked here; the write
flows through the symlinks into the canonical store, and every other Laravel
project whose profile includes those skills sees the update immediately.

**Verify symlinks survive each `boost:update`** ‚Äî boost may replace a symlink
with a real folder when it reinstalls a skill:

```bash
# in the Laravel project, after boost:update:
ls -la .agents/skills/ | grep laravel     # must show "->" (symlink)
# if a skill shows as a real folder (no arrow), restore the link:
cd /path/to/agent-brains && ./setup.sh /path/to/laravel-project laravel
```

If boost installs a brand-new skill, add it to `profiles/laravel.list` (or the
relevant profile) and re-run `./setup.sh /path/to/project laravel`.

## Profiles

A profile is a plain-text manifest in `profiles/<name>.list` ‚Äî one skill name
per line (matching the folder/file name in `.agents/skills/`). A manifest can
pull in another with `@include <name>`, so common skills are defined once.

```
# profiles/laravel.list
@include frontend        # pulls in frontend.list, which pulls in common.list
laravel-best-practices
laravel-permission-development
livewire-development
volt-development
pest-testing
```

| Profile   | Includes            | Use for                                  |
| --------- | ------------------- | ---------------------------------------- |
| `laravel` | frontend ‚Üí common   | PHP full-stack (Blade + backend)         |
| `nextjs`  | frontend ‚Üí common   | Next.js / React                          |
| `astro`   | frontend ‚Üí common   | Astro content/static sites               |
| `nodejs`  | common              | Node.js backend APIs (no UI skills)      |
| `full`    | (whole-dir symlink) | Escape hatch ‚Äî every skill, zero curation |

### Adding / editing a profile

1. Edit `profiles/<name>.list` (create it for a new profile).
2. Re-run `./setup.sh /path/to/project <name>` to refresh the project's symlinks.

### Making a skill available everywhere

Add it to `profiles/common.list` ‚Äî every profile that `@include common` (all of
them) will pick it up on the next `./setup.sh`.

### Adding a new skill ‚Äî canonical workflow

When you create a new skill folder under `.agents/skills/<name>/`, target
projects will **not** see it until you do both steps below. Do not create
symlinks manually inside project `.agents/skills/` directories ‚Äî `setup.sh`
deletes and regenerates symlinks from the manifests (line 121 of `setup.sh`),
so any hand-crafted symlink is wiped on the next run.

1. **Author the skill** in `.agents/skills/<name>/SKILL.md` (this repo).
2. **Register it in the manifest(s)**:
   - Generic / cross-stack skill ‚Üí `profiles/common.list` (inherits to every profile)
   - Frontend / motion / UI skill ‚Üí `profiles/frontend.list` (laravel, nextjs, astro, full ‚Äî not nodejs)
   - Stack-specific skill ‚Üí the relevant `profiles/<stack>.list`
   - For documentation only, also add a line to `profiles/full.list` (the `full` profile is special-cased in `setup.sh` to symlink the whole dir, but `full.list` is kept as a reference of what "everything" contains).
3. **Re-run setup.sh on each project** that should pick it up, with its profile:
   ```bash
   ./setup.sh /path/to/project laravel   # or nextjs | astro | nodejs | full
   ```
   Idempotent: wipes and recreates only symlinks, never touches real files.

To re-sync many projects at once, see the fingerprint trick: resolve each
profile with `resolve_manifest` (defined in `setup.sh`) and match against the
sorted list of symlinks present in `project/.agents/skills/` ‚Äî that recovers
which profile each project uses without a registry.

## Recommended Workflow

### Project setup

1. **`setup`**: Interrogates you to bootstrap the project and create `.ai/context/GLOSSARY.md`.

### Feature lifecycle (pre ‚Üí during ‚Üí post)

2. **`feature`**: collaborative design with a HARD-GATE ‚Äî no code until a PRD is presented and approved. Produces a PRD at `.ai/features/[name]/prd-*.md` and a TDD task list at `.ai/features/[name]/tasks-*.md`.
   - optional: **`grill-with-docs`** to stress-test the PRD against the glossary/ADRs.
3. **Framework agents** (`/laravel`, `/nextjs`, `/astro`): implement the task list task-by-task using **`tdd`** (red-green-refactor). Switch to a fresh chat and reference the task file path.
   - optional: **`to-tickets`** to publish each task as a tracer-bullet issue on the tracker.
   - optional: **`to-spec`** if a formal, shareable PRD is needed on the tracker.
4. **`verification-before-completion`**: before claiming "done", run the actual lint/typecheck/test commands and confirm the output.

### Cross-cutting

5. **`diagnosing-bugs`**: auto-activates on bugs ‚Äî build a reproducible case first, then fix.
6. **`improve-codebase-architecture`**: runs periodically to produce before/after refactoring reports.
7. **`/handoff`**: when a chat becomes too long (deep context, many back-and-forths), compact the conversation into a doc the next agent can pick up in a fresh chat. Saved to `$TMPDIR` by default; `--keep` to persist it in `docs/handoffs/`.

### Frontend & Motion

The frontend skills are split by **intent**, not by "frontend" in general. Triggering the wrong one is the most common cause of generic AI-slop output ‚Äî each owns a specific job and points to its siblings when the request is out of scope.

| Intent | Skill |
| --- | --- |
| Bold, distinctive aesthetics ‚Äî landing pages, marketing, hero sections, concept UI, posters, portfolios. Anti-AI-slop. | `frontend-design` |
| Product UI structure & design systems ‚Äî dashboards, admin panels, SaaS, charts. Database of 161 color palettes, 57 font pairings, 25 chart types. | `ui-ux-pro-max` |
| Animations, transitions, easing curves, spring physics, micro-interactions, gestures, polish/feel. Emil Kowalski (Sonner/Vaul). | `emil-design-eng` |
| shadcn/ui components (add, fix, style, compose). | `shadcn` |
| Tailwind utility classes, responsive layouts, dark mode. | `tailwindcss-development` |
| Scalable design systems with Tailwind v4 (tokens, component libraries). | `tailwind-design-system` |
| View Transition API ‚Äî page transitions, shared-element animations, directional navigation. | `vercel-react-view-transitions` |
| React composition patterns (compound components, render props, context). | `vercel-composition-patterns` |
| React/Next.js performance (RSC, data fetching, bundle size). | `vercel-react-best-practices` |
| **Review** UI against Vercel Web Interface Guidelines. | `web-design-guidelines` |
| **Review** animation/motion code (slash command, brutal craft bar). | `/review-animations` |

### Anti-patterns

- Going straight to a framework agent (`/laravel`, `/nextjs`, ‚Ä¶) without a PRD/task list ‚Üí use `/feature` first.
- Skipping TDD on individual tasks ‚Üí every task goes through `tdd`.
- Letting a chat grow too long without `/handoff` ‚Üí compact and resume fresh.
